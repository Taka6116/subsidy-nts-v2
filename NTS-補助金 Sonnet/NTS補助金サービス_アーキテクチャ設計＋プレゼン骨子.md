### 1-2. AWSサービスマッピング

| レイヤー | コンポーネント | AWSサービス | 理由 |
|---------|------------|-----------|------|
| **フロントエンド** | 提携先ポータル | CloudFront + S3（React SPA） | 静的ホスティング。低コスト・高速配信 |
| **フロントエンド** | ゲートLP / 診断ツール | CloudFront + S3 or Amplify | LP・診断はWordPressとは別で管理。A/Bテストしやすい |
| **API** | APIレイヤー | API Gateway + Lambda | サーバーレス。初期コストゼロ。スケーラブル |
| **認証** | 提携先ログイン | Cognito | メールベースの認証。提携先ごとのアクセス制御 |
| **DB** | 補助金制度DB | DynamoDB | 制度情報はスキーマレスで変動が多い。NoSQL向き |
| **DB** | 紹介案件管理 | RDS（PostgreSQL） | リレーショナルデータ。案件→提携先→顧客の関係管理 |
| **通知** | 新制度通知・案件進捗 | SNS + SES | メール＋プッシュ通知。提携先への即時配信 |
| **分析** | ダッシュボード用データ | QuickSight or 自前React | 提携先が紹介実績・フィーを確認する画面 |
| **AI連携** | 速報記事生成 | Bedrock（Claude）or 外部API | NASシステムとの連携。案Aのバックアップ用 |
| **ストレージ** | 営業資料テンプレート | S3 | PDF/PPTX の保管・配信 |
| **CI/CD** | デプロイ | CodePipeline + CodeBuild | 自動デプロイ |
| **監視** | ログ・アラート | CloudWatch | 異常検知・利用状況モニタリング |

### 1-3. 提携先ポータルの4つの核機能

#### ① 商材×補助金マッチングチェッカー

**目的：** 提携先が「自分の商材に使える補助金があるか」を即座に確認できる

**仕組み：**
- 提携先が自社商材を登録（カテゴリ選択式：IoT機器、ロボット、ソフトウェア等）
- 登録された商材に対して、現在申請可能な補助金制度が自動マッチング
- 結果表示：「この商材は省力化補助金の対象です。補助率：最大1/2、上限1,500万円」

**提携先の体験：**
「うちのロボットアーム、補助金使えるのか…使える。じゃあ顧客に提案できるな」→ 紹介へ

**技術実装：**
- 商材カテゴリ × 補助金要件のマッチングロジック（Lambda関数）
- 制度情報はDynamoDBで管理。公募開始/終了を自動更新
- 将来的にはAIで商材説明文からの自動マッチングも可能

---

#### ② 共有用 簡易診断ツール

**目的：** 提携先が自分の顧客（エンドユーザー）に送れる診断ツール

**仕組み：**
- 提携先ごとにユニークURLが発行される（例：`nts.co.jp/check?partner=ABC`）
- エンドユーザーが3〜5問に回答（業種、従業員数、投資予定額、課題）
- 結果：「あなたの会社に使える可能性のある補助金があります」→ NTSの相談フォームへ
- 提携先のダッシュボードに「誰が診断を完了したか」が表示される

**提携先の体験：**
「お客さん、ちょっとこれ試してみてください」とURLを送るだけ → リードが自動でNTSに流れる → 提携先は何もしなくていい

**技術実装：**
- React SPA。CloudFront配信
- パートナーIDをクエリパラメータで追跡（UTMと同等）
- 診断結果 → API Gateway → Lambda → RDSに保存
- Cognito認証済みの提携先ダッシュボードからリード一覧を確認

---

#### ③ 紹介管理ダッシュボード

**目的：** 提携先が紹介した案件のステータスと報酬を一目で確認できる

**表示内容：**
- 紹介案件一覧（企業名、申請補助金、ステータス）
- ステータス：相談中 → 申請準備中 → 申請済み → 採択 → 入金完了
- 紹介フィーの確定額・入金予定日
- 月次/年次の紹介実績サマリー

**提携先の体験：**
「先月紹介した3社のうち2社が採択された。紹介フィー合計45万円、来月入金か」→ 信頼感 → さらに紹介

**技術実装：**
- RDS（PostgreSQL）で案件テーブル管理
- 提携先ごとのビューをCognito認証でアクセス制御
- ステータス変更時にSESでメール通知

---

#### ④ 営業資料テンプレートライブラリ

**目的：** 提携先が自分の営業シーンでそのまま使える資料を提供

**コンテンツ例：**
- 「補助金が使えます」チラシ（提携先ロゴ差し替え可能）
- 省力化補助金の概要1枚もの
- 事業承継補助金の概要1枚もの
- 顧客向け提案書テンプレート（商材+補助金のセット提案用）
- 補助金FAQ集（顧客からの想定質問と回答）

**提携先の体験：**
「来週の商談で使おう。PDF落としてロゴだけ変えれば完成だ」→ NTSの存在が提携先の営業に溶け込む

**技術実装：**
- S3にPDF/PPTX格納
- ダッシュボードからダウンロード
- 将来的にはCanva API連携でオンライン編集も可能

## Part 2：コンセプト＋タグライン

---

### 2-1. コンセプトの構造

**サービスの本質：**
NTSの補助金サービスは「申請代行」ではなく「提携先の営業を強くする仕組み」。
提携先が自分の顧客に「補助金が使えますよ」と自信を持って言える状態を作ることで、
提携先の成約率が上がり、NTSにはリードが流れ、エンドユーザーは補助金で得をする。三方よしの構造。

**コンセプトキーワード：**
「あなたの営業が、補助金で強くなる。」

これはNTSを「補助金の会社」ではなく「提携先のビジネスを強化する会社」として位置づける。

---

### 2-2. タグライン（2系統）

#### 提携先向け（メイン軸）

**第一候補（推奨）：**
> **「補助金が使えます」——その一言が、御社の武器になる。**

理由：提携先の営業現場で実際に起きるシーンを描写。抽象概念ではなく行動を想起させる。

**第二候補：**
> **御社の提案に、補助金という選択肢を。**

理由：より落ち着いたトーン。士業向けにはこちらの方が刺さる可能性。

**第三候補：**
> **紹介するだけ。あとは私たちが動きます。**

理由：提携先の「何もしなくていい」安心感を前面に。紹介ハードルを徹底的に下げる。

---

#### エンドユーザー向け（補助軸）

**第一候補（推奨）：**
> **その設備投資、補助金の対象かもしれません。**

理由：「気づかせる」を体現。知らなかった人に「え？」と思わせる訴求。

**第二候補：**
> **人手不足も、事業承継も。使える補助金があります。**

理由：対象領域を明示。自分ごと化しやすい。

**第三候補：**
> **補助金の選定から申請、採択後の実行まで。まず話を聞かせてください。**

理由：前回資料の「寄り添い」コンセプトを残しつつ、対応範囲を具体化。

### プレゼンテーション概要
- **クライアント：** 株式会社日本提携支援（NTS）
- **対象者：** 桜庭氏、大野氏（経営層・意思決定者）
- **目的：** 補助金サービスの集客アーキテクチャを提案し、開発方針のGo判断を得る
- **文脈：** 第四回定例会のフォローアップ。前回議論を踏まえた方向転換の提案
- **推奨所要時間：** 20分
- **スライド枚数：** 12枚

### ストーリー構造：B（機会提示型）を採用

理由：前回の定例会で「概念ではなくアーキテクチャから」という方向転換が合意された。これは新しい戦略の提案であり、「やらないリスク」と「やるなら今この形で」を明確に示す必要がある。

### 全体ストーリーライン

「補助金は概念では売れない」→ 競合を見てもLP・コピーで差がつかない現実 → ユーザーは補助金を検索しないという行動特性 → だからこそ攻めるべきは提携先チャネル → NTSには提携先ネットワーク構築の意図が既にある → それを仕組み化する「提携先イネーブルメント」というアーキテクチャ → 具体的なプロダクト設計 → 開発ロードマップ → Go判断

#### スライド 3：補助金市場は「概念」では勝てない——その構造的な理由

- **メッセージ：** 補助金のLPは9割が同じ訴求。概念の差別化は構造的に不可能
- **ビジュアル：** 競合LPのスクリーンショット3〜4枚を並べて類似性を視覚化（前回資料のp6を再利用）
- **根拠データ：**
  - 「採択率」「件数」「成功報酬型」「丸ごとサポート」——どこも同じ
  - コピーを変えてもサービス自体が同質
- **スピーカーノート：** 「ここに我々がどんなに良いコピーを書いても、根本的には同じ土俵に立つことになります」

---

#### スライド 4：ユーザーは「補助金」を検索しない——啓蒙が必要な市場

- **メッセージ：** 補助金はM&Aと違い、ユーザーが自発的に探しに来ない。「気づき」の領域
- **ビジュアル：** 2×2マトリクス（横軸：ニーズの顕在度、縦軸：検索行動の有無）
  - M&A：顕在×検索あり → LP直接集客が有効
  - 補助金：潜在×検索なし → 啓蒙＋チャネル設計が必要
- **根拠データ：**
  - 「補助金を使いたい人は自分の税理士に相談する」（吉田氏の指摘）
  - 補助金は「目的」ではなく「手段への気づき」
- **スピーカーノート：** 「御社のM&Aオファーは独自性があるからLPで直接集客できる。しかし補助金はそうではない。だからこそ、別のアプローチが必要です」

---

#### スライド 5：攻めるべきは提携先チャネル——NTSが唯一勝てる構造

- **メッセージ：** エンドユーザー直接集客ではなく、提携先を「NTSの営業部隊」にする構造が最適解
- **ビジュアル：** フロー図
  ```
  提携先（ベンダー・士業）
    → 自社顧客に「補助金使えますよ」と一言
    → エンドユーザーがNTSに相談
    → NTSが申請サポート
    → 採択 → 紹介フィーが提携先に還元
    → 提携先がさらに紹介 [循環]
  ```
- **根拠データ：**
  - 御社テキスト「提携先獲得を非常に重視」
  - 競合は補助金専業。NTSだけがM&A×補助金の文脈を持ち、提携先が「経営の相談先」として紹介できる
- **スピーカーノート：** 「なぜNTSなのか。それは補助金の専門家だからではなく、M&Aの知見を持つ経営支援のプロだから。提携先が自分の顧客に安心して紹介できる理由がここにあります」

---

#### スライド 6：提携先の本音——4つの壁を仕組みで壊す

- **メッセージ：** 提携先が紹介に踏み切れない4つの壁に、プロダクトで答える
- **ビジュアル：** 4象限（壁 → 解決策の対比表）

| 提携先の壁 | プロダクトでの回答 |
|-----------|----------------|
| 「自社商材に補助金が使えるか分からない」 | 商材×補助金マッチングチェッカー |
| 「顧客にどう説明すればいいか分からない」 | 営業資料テンプレートライブラリ |
| 「紹介した後どうなるか見えない」 | 紹介管理ダッシュボード |
| 「紹介して変なことになったら怖い」 | 案件ステータスの可視化＋NTSの対応品質 |

- **スピーカーノート：** 「前回資料で整理した提携先の課題を、コンセプトではなくプロダクトで解決します」

- **ビジュアル：** 2画面のモックアップイメージ（左：チェッカー画面、右：診断ツール結果画面）

- **スピーカーノート：** 「提携先がやることは2つだけ。チェッカーで確認して、診断ツールのURLを顧客に送る。あとはNTSが全部対応します」

#### スライド 9：核機能③④ ——紹介管理＋資料ライブラリ

- **メッセージ：** 紹介後の不安を可視化で解消し、営業現場で使える武器を常備する
- **ビジュアル：** ダッシュボードのワイヤーフレーム（ステータス一覧＋フィー確認画面）
- **補足：**
  - ダッシュボード：紹介案件のリアルタイムステータス＋紹介フィー確定額
  - 資料ライブラリ：チラシ、概要資料、提案書テンプレ、FAQ集
- **スピーカーノート：** 「特にダッシュボードの透明性が重要です。提携先が"紹介したら放置された"と感じた瞬間に関係が終わります。見える化が信頼の基盤になります」

## Part 5：ベター案の提示

### 提案①：ゲートLPの分岐をなくす選択肢

現在の設計はゲートページで「企業向け」「提携先向け」に分岐するが、実はゲートページ自体が不要かもしれない。理由は、提携先向けLPはエンドユーザーの目に触れる必要がないため。別URL（例：`partner.nts.co.jp`）で完全に分離し、ゲートLPはエンドユーザー専用にする方がシンプルで、かつ提携先向けLPにはNTSが直接案内できる。

**判断基準：** SEOでオーガニック流入を狙うならゲートページ型。提携先への直接営業で獲得するならURL分離型。後者の方が現実的だと思われるが、Takaさんの判断に委ねる。

### 提案②：Phase 0にLP・診断だけでなく「提携先向け1枚ものPDF」を含める

ポータルが完成する前でも、提携先獲得は始められる。Phase 0で「提携先向けの案内PDF」を作成し、桜庭氏の既存ネットワークに直接配布する。これでポータル開発と提携先開拓を完全に並行で走らせられる。

### 提案③：FIRST GATEの名称をポータル側に移す

エンドユーザー向けのサービス名称ではなく、提携先ポータルの名称として「FIRST GATE」を使う方がコンセプトに合う。提携先にとってNTSが「最初の入り口」になるという意味で。エンドユーザー向けには、NTS本体ブランドで十分。
