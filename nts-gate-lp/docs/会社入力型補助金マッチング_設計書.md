# 会社入力型 補助金マッチング — 設計書

**版:** 0.1（ドラフト）  
**前提:** Next.js アプリ（`nts-gate-lp`）、AWS（RDS PostgreSQL、Lambda、Amazon Bedrock Claude 等）、補助金データは **jGrants API** を正とする。

---

## 1. 目的・スコープ

### 1.1 プロダクトゴール

- **エンドユーザー向け:** 「自社が対象になり得る補助金」を、**会社名（必須）＋業種**を起点に提示する。さらに **AI により**、各制度の範囲内で「何に使えるか・何を導入できるか・何が実現し得るか」を**例示ベース**で文章化する（断言はしない）。
- **提携向け:** 補助金そのものの宣伝ではなく、**自社のビジネス課題解決の手段として補助金提案が機能する**文脈で訴求。想定サービスは **省力化補助金・事業承継（引継ぎ）補助金**の申請サポート特化、手付金＋成功報酬型の料金体系を反映する。

### 1.2 現行からの変更（方針）

| 項目 | 現行 | 変更後 |
|------|------|--------|
| 入力 | 診断ウィザード（質問フロー） | 会社名（必須）＋業種（選択または補助）※URL は Phase2 で任意 |
| 法人の特定 | なし | 法人番号系 API で検索 → **複数候補時はユーザーが1社を選択** |
| 制度データ | クライアント側スコアリング等 | **jGrants API 由来データを DB に同期**しマッチング・表示に利用 |
| 結果の深さ | 制度カード中心 | 制度情報に加え **AI による利用イメージ**（根拠は同期データに限定） |
| 提携導線 | ヘッダー等 | エンドユーザー LP では **CTA は控えめ**。**「税理士・士業の方へ」相当の独立セクション**から提携 LP へ |
| 提携 LP | 外部 `partner.firstgate.jp` 等 | **同一 Next.js アプリ内 `/partner` を推奨**（エンジン共有・保守一元化）。ドメイン要件があれば別途ホスト・リバプロで調整 |

### 1.3 スコープ外（本書では詳細未固定）

- **ImpactNumber（最大金額）セクション:** 残置／刷新／廃止は別途決定。
- **URL 入力の本格利用:** Phase2（検索ヒント・候補絞り込み）。

## 2. ユーザー体験（エンドユーザー）

### 2.1 フロー

1. **オープニング（スプラッシュ）** — **残す**（現行 `HomeEntrance` 相当）。
2. LP 上で **会社名入力（必須）**、**業種選択（必須。補助入力があれば設計で定義）**。
3. **法人検索** → ヒット0件／1件／複数件。
   - **複数件:** 候補一覧から **1社を明示選択**（必須）。ここまでを同一画面内で完結させる。
4. 確定した法人情報（法人番号・商号・所在地等、API 取得範囲）＋業種をもとに **マッチング実行**。
5. **同一ページ内**でローディング → **結果表示**（制度一覧カード ＋ 各制度または束ねての **AI 利用イメージ**）。
6. **無料相談・問い合わせ CTA**（文言・遷移先は要件に合わせる）。

### 2.2 コピー・法務表現

- **「個人情報は不要」** の趣旨は維持しつつ、**会社名・法人識別に関する利用**について一言説明する（広告表示との整合）。
- AI 出力には必ず **「例・イメージ」「公募要領・最新情報で確認」** 等のディスクレーマを付与する。

### 2.3 LP 構成（案）

- **ヘッダー:** 提携導線は現状程度に抑制可。
- **メイン CTA ブロック:** 会社名＋業種入力（新コア）。
- **士業・税理士向けセクション:** 「クライアントに補助金提案できていますか？」等の文脈 → **提携 LP（`/partner`）へ CTA**。
- **配置の推奨:** **Trust（会社ストーリー）の直前**に士業セクションを置く案をデフォルトとする。Impact セクションを残す場合は **`Impact → 士業 → Trust`** の順が読みやすい。

## 3. ユーザー体験（提携 `/partner`）

### 3.1 方針

- **エンドユーザーと同一マッチングエンジン**（API・DB・AI ロジック共有）。
- UI 文言は **「顧客企業の会社名を入力」** ニュアンスに寄せる。
- 訴求: **補助金目当てではなく、提案・成約・顧客価値のための手段**として補助金を位置づける。主力制度（**省力化・事業承継／引継ぎ**）と **手付金＋成功報酬** をコンテンツに反映。

### 3.2 機能

- **対象補助金表示機能を持つ**（合意）。問い合わせのみにしない。

## 4. 機能要件

### 4.1 入力

| 項目 | 必須 | 備考 |
|------|------|------|
| 会社名 | 必須 | 法人番号 API の商号検索に使用 |
| 業種 | 必須 | 選択 UI。業種マスタは jGrants／社内定義と整合させる |
| 公式サイト URL | 任意（Phase2） | Phase1 では UI に載せない or 非活性でも可。「公式サイトが分かる方は」文言は Phase2 解禁時に |

### 4.2 法人の確定

- 検索結果 **複数件時は必ずユーザーが1社を選択**（自動確定しない）。
- 0件時は再入力案内または手動補助（将来拡張）。

### 4.3 マッチング

- **データソース:** PostgreSQL に同期済みの jGrants 由来データ（下記粒度）。
- **ロジック:**  
  - **構造化フィルタ**（業種・地域・規模等、データに存在する属性）を優先。  
  - **LLM** は「フィルタ後の候補＋法人コンテキスト」に対し、**利用イメージ文の生成**に限定。制度要件の捏造を禁止し、**入力した構造化フィールドのみ**を根拠に記述する。

### 4.4 表示データ粒度（jGrants 同期）

最低限想定する項目（API／実データに合わせてマッピング表を別紙化する）:

- 制度名  
- 上限額（表示用）  
- 申請期限（表示用。ソースは同期データ）  
- 対象業種（コードまたはテキスト）  
- 対象用途・対象経費の要約  

### 4.5 AI 出力（Claude / Bedrock）

- **入力:** 確定法人情報、ユーザー選択業種、各候補制度の同期済みフィールド（JSON 等）。  
- **出力例:**  
  - その制度で **想定しうる導入・投資の例**  
  - **業務上実現しうること**（例示）  
- **禁止:** 採択確約、金額確定、要領にない内容の創作。

### 4.6 結果画面の遷移

- **同一ページ内**での段階表示を MVP とする（合意）。  
- 将来: 共有用 URL（クエリまたはトークン）を検討可。
